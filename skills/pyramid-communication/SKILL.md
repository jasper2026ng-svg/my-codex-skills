# Pyramid Communication

Use this skill to make communication easier to understand and act on. It converts scattered thoughts into an answer-first storyline with clear support, logical grouping, and a concrete ask.

Use `structured-problem-framing` first if the problem is still unclear. Use `critical-questioning` first if the core claim is weak or untested. Use this skill after the user has at least a tentative conclusion, recommendation, or message goal.

Answer-first is a structural principle, not a rule that every message must announce the conclusion in its first sentence. For a resistant or unfamiliar audience, use brief situation and complication context when needed, but make the answer explicit early and do not bury it.

Do not summarize the source book or quote long passages from it. Treat the book-derived material as method inspiration, and write in your own words.

## Workflow

1. Clarify the communication task.
   - Identify the audience, decision context, desired action, channel, length, and deadline.
   - If missing, infer practical defaults and label them as assumptions.

2. Extract the top answer.
   - State the one sentence the audience should remember.
   - Make it a claim, recommendation, or decision point, not a topic label.
   - Avoid empty headings such as "background," "issues," or "next steps" unless each contains a real point.

3. Build the audience question.
   - Infer what question the audience is asking after hearing the top answer.
   - Use that question to decide what must come next.
   - If the audience likely asks multiple questions, sequence them in decision order.

4. Group supporting ideas.
   - Group points by the same kind of logic: reasons, causes, options, actions, risks, timeline, stakeholders, or metrics.
   - Keep each group internally consistent.
   - Remove points that do not support the top answer or move them to appendix/context.

5. Check horizontal logic.
   - Within each group, order points by one clear logic: time sequence, structure, degree of importance, cause-effect, or mutually exclusive options.
   - Avoid mixing categories such as cause, action, and evidence in the same level.

6. Write real summaries for each group.
   - Each group heading must express the insight produced by its children.
   - Do not use label-only headings like "three problems" or "key actions" if a sharper claim is possible.
   - If you cannot write the summary sentence, revisit the grouping.

7. Add situation-complication-question when useful.
   - Use this for proposals, updates, and reports where the audience needs context.
   - Situation: what stable context everyone accepts.
   - Complication: what changed, broke, or now requires attention.
   - Question: what decision or response is needed.
   - Answer: the top recommendation.

8. Produce the final communication.
   - Choose the form that fits the user's need: short message, executive summary, meeting readout, memo outline, or slide storyline.
   - Keep language direct, specific, and operational.
   - End with the requested action, decision, or next step.

## Quality Bar

A good output should:

- put the answer before the explanation;
- adapt the opening sequence to the audience without burying the answer;
- make every heading a real claim, not a topic label;
- group ideas by one clear logic at each level;
- avoid dumping unrelated facts into the same branch;
- make support mutually distinct enough for practical use;
- remove or demote details that do not support the top answer;
- fit the user's operating context, especially HRBP, sales, management meetings, boss updates, and cross-cultural execution.

## When Not To Use

Do not use this skill when:

- the user is still exploring and does not have a tentative conclusion;
- the task is mainly creative writing or emotional expression;
- the core reasoning has not been checked and may be wrong;
- the user needs a deep analysis rather than communication structure;
- the message is legally sensitive and requires exact wording review.

## 目录与公共划线共同指向的核心

1. 对特定的人而言，问题表现为其感知到的现状与期望状态之间的差距；
2. 不要把已有解决方案当成问题定义；
3. 同一情境可以有多个问题定义，且无法最终确认某个定义绝对正确；
4. 需要识别是谁遇到了问题、谁应该解决、谁会承受后果；
5. 应从不同层次和不同利益相关者视角重述问题；
6. 需要判断当事人是否真的希望问题被解决，以及现状让谁受益；
7. 每个解决方案都可能制造新问题，应检查二阶影响；
8. 不要无限追求完美定义，问题清楚到足以指导下一步时就应行动和继续校准。

## 读者反馈揭示的价值与风险

推荐反馈主要认可：暂停过早解题、转换视角、识别问题归属、检查解决方案副作用。

负面反馈高度集中在：

- 寓言与故事较多，方法没有被显性整理；
- 部分读者认为中文翻译拗口；
- 观点启发性强，但缺少严密证据和完整执行步骤；
- 读者容易只记住金句，却难以形成稳定工作流程。

微信读书当前数据中共有 466 个评分记录，其中 318 个为好评、114 个为一般、34 个为差评；111 位资深会员中的推荐比例为 68.5%。这说明总体评价偏正面，但并非没有明显争议。

## 可转 Skill 性

结论：**部分适合。**

适合提取：

- 触发条件与认知警报；
- 问题归属和利益相关者检查；
- 多种暂定问题定义；
- 是否值得解决、是否真的想解决；
- 解决方案副作用检查。

需要其他来源或 Agent 综合：

- 标准化输入字段；
- 可重复执行的步骤顺序；
- 证据阈值与质量标准；
- 输出模板；
- 与结构化拆解、用户研究和决策审查的交接规则。

因此，本书应被定位为 `real-problem-discovery` 的概念主锚点，而不是唯一方法来源。

## 当前 Skill 的来源分层

### 书中明确

- 现状感知与期望之间的差距；
- 谁的问题、谁来解决、问题来自哪里；
- 不把解决方案当作问题定义；
- 问题定义始终暂定；
- 是否真的想解决、解决后会产生什么新问题。

### Agent 综合

- 原始信号、场景、成本和约束的标准字段；
- 生成三种替代问题表述；
- 明确下一步 Skill 路由；
- 输出模板和置信度；
- 将问题清晰度转成可检查的质量门槛。

### 用户经验

- 难验证的人性判断不适合作为第一行动切口；
- 优先选择可观察、可改变、可复盘的系统变量；
- 不验证所有假设，而是根据决策影响和错误代价选择验证。

### 待验证

- 这套组合流程能否在不同管理、HRBP 和业务场景中稳定提升问题定义质量；
- 三种替代问题表述是否总有价值，还是会在低复杂度任务中增加冗余；
- 与 `structured-problem-framing` 的交接点是否足够清楚。

## 本次修正结论

- 不再声称 Skill 能找到唯一、最终的“真问题”；
- 输出改为“当前证据下更好的暂定问题定义”；
- 明确加入期望、现状感知、问题归属、解决意愿和二阶影响；
- 保留用户提出的可观察系统变量原则，但标注为用户经验，不冒充原书结论；
- Skill 继续保持早期状态，等待真实案例与延迟复测。

## 独立前向测试

第一次独立测试能够区分事实、解释和预设方案，也能降低因果判断的置信度；但仍然过早给出了完整改进动作，只生成了一个问题定义，并跳过了利益相关者的现状与期望差距。

据此新增停止规则：发现阶段第一轮必须给出至少三个暂定问题定义，只允许提出一个最小证据检查，不得直接进入完整方案设计。

第二次独立测试换成新人离职与培训场景后，仍然直接生成了两周诊断、五项行动和指标体系。说明原 Skill 同时承担发现、诊断、行动设计和路由，边界过宽，仅增加提示无法解决。

据此将 Skill 收窄为强制的发现阶段合同：分类原始信号、识别利益相关者差距、提出三个暂定定义、选择工作定义、提出一个最小检查，然后停止并交给下游 Skill。

第三次独立测试能够提出三个不同问题方向，也能识别“重做看板”只是方案，但仍给出四项验证与局部改造。由此将第一轮停止规则同时写入 Skill 的触发描述和默认提示，确保 Agent 在加载正文前就看到边界。

目前不把三次测试判定为“通过”。第三次测试后更新的触发描述和默认提示尚需在新的真实场景中复测；在复测通过前，Skill 保持早期版本，并保留 Agent 可能过早进入方案设计的风险提示。
